## Security Findings

**CRITICAL** `backend/app/services/image_service.py:49` ‚Äî After `img.verify()` the image is opened a second time via `Image.open(io.BytesIO(data)).format`. The `verify()` call on line 44 consumes the internal file pointer; opening a second instance is correct here, but `verify()` is documented to destroy the image object ‚Äî the second `Image.open` is therefore safe, but only because the bytes buffer is re-used. However, the extension whitelist (`_FORMAT_TO_EXT`) only allows JPEG, PNG, WEBP. GIF and BMP would return `None` from `.get()` and raise a `ValueError`. This is the correct and intended behavior ‚Äî documenting it explicitly would help maintainers.
Standard: File upload validation (MIME type, size limits).
Assessment: CORRECT ‚Äî MIME validation via magic bytes is properly implemented. No change required. Consider adding a comment explaining why `verify()` + a second `Image.open` is the correct pattern.

**HIGH** `docker-compose.yml:38` ‚Äî Backend is started with `--reload` flag via `command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload`.
Standard: Dev-only flags not in production.
Assessment: The `docker-compose.prod.yml` override correctly removes `--reload` (line 7). However the dev compose file is the default used by `make up`, and `--reload` causes uvicorn to watch the entire filesystem at `/app`. If the production deployment ever accidentally uses the dev compose file, `--reload` would be active. The existing prod override mitigates this as long as operators remember to use it.
Fix: Add a comment on the `command` line in `docker-compose.yml`: `# DEV ONLY ‚Äî overridden in docker-compose.prod.yml`. This is a documentation fix, not a code change.

**MEDIUM** `docker-compose.prod.yml:7` ‚Äî Uses `--workers 1` with a comment acknowledging the in-memory session store limitation. This is documented and intentional, but the production single-worker constraint is a reliability risk under load.
Standard: Production configuration.
Fix: The file already documents the TODO: replace with Redis-backed session store. Track this as a known architectural limitation. No immediate action.

**MEDIUM** `backend/app/core/config.py:16` ‚Äî `cors_origins` defaults to `"http://localhost:5173,http://localhost:3000"`. In production this default will be used if `CORS_ORIGINS` is not set in the environment, allowing localhost origins in prod.

## DevOps Findings

**HIGH** `frontend/Dockerfile:24` ‚Äî Runtime stage uses `CMD ["npm", "run", "dev", "--", "--host"]` which starts the Vite dev server. The production override in `docker-compose.prod.yml` replaces this with `npx serve dist -l 5173`, but the built image's default CMD remains the dev server.
Standard: Dev-only flags not in production.
Fix: Change the Dockerfile `CMD` to the production command: `CMD ["npx", "serve", "dist", "-l", "5173"]`. The dev compose file can override this with `command: npm run dev -- --host`. This makes the image safe to `docker run` without compose.

**MEDIUM** `backend/Dockerfile` ‚Äî Multi-stage build, pinned `python:3.12.8-slim-bookworm`, non-root user (`app`), `.dockerignore` present. All Docker best practices are followed.
Standard: Docker best practices.
Assessment: PASS.

**MEDIUM** `frontend/Dockerfile` ‚Äî Multi-stage build, pinned `node:20.20.1-alpine3.23`, non-root user (`app`). However, the runtime stage copies the entire builder output including `node_modules/` (devDependencies) because `COPY --from=builder /app .` copies everything. For production serving, only `dist/` and the serve binary are needed.
Standard: Multi-stage builds.
Fix: In the runtime stage, `COPY --from=builder --chown=app:app /app/dist ./dist` and install only `serve` via `npm install -g serve`. This reduces the production image size significantly.

**MEDIUM** `docker-compose.yml:44` ‚Äî Frontend service sets `VITE_API_BASE_URL=http://localhost:8000`. During Docker build, Vite inlines this at compile time. But the frontend container binds to port 5173 with hot-reload, so the env var is passed at runtime to the dev server ‚Äî this works in development but would be wrong in production builds where the env var must be baked in at `npm run build` time.
Standard: DevOps configuration.
Assessment: This is a known dev-vs-prod distinction. For production, the builder stage in the Dockerfile must receive `VITE_API_BASE_URL` as a build-arg. The current setup does not expose a build-arg mechanism in the Dockerfile or Compose prod override.
Fix: Add `ARG VITE_API_BASE_URL` and `ENV VITE_API_BASE_URL=$VITE_API_BASE_URL` to the frontend Dockerfile builder stage, and pass it via `docker-compose.prod.yml` `build.args`.

**MEDIUM** `Makefile:62-64` ‚Äî `rebuild-db` target calls `docker compose stop backend db` and `docker compose start backend`, but does not run `alembic upgrade head` after restarting. Compare with `reset-db` which does run migrations. A developer running `make rebuild-db` may end up with a running backend against an out-of-date schema.
Standard: DevOps consistency.
Fix: Add `@docker compose exec backend alembic upgrade head` at the end of the `rebuild-db` target, matching the `reset-db` pattern.

**LOW** `docker-compose.yml` ‚Äî No `frontend` healthcheck is defined. The backend has a proper healthcheck. The frontend being healthy is less critical but inconsistent.
Standard: DevOps best practices.
Fix: Minor; add a `healthcheck` to the frontend service testing port 5173.
