# Alembic

> Last updated: 2026-04-09
> Source: Discovered during feature 055 (program tree area grouping).

## Quirks & Gotchas

### `alembic revision --autogenerate` detects drift beyond your intended changes

Alembic diffs your SQLAlchemy models against the live DB schema and generates migration ops for **every** difference — not just the ones your feature introduced. In feature 055, the autogenerated migration for a tree-grouping change also included:

- Dropping `ondelete='SET NULL'` from `programs.agency_id` (unrelated, previously hand-edited)
- Recreating `fk_teams_lead_id` with a different constraint name

Both would silently alter existing production constraints if committed as-is.

**Rule**: **always review and trim autogenerated migrations**. Remove operations that don't relate to the feature. Run `alembic upgrade head --sql` to preview the actual DDL before applying.

### `drop_constraint(None, ...)` fails at runtime

Alembic uses `None` for auto-generated FK constraint names when it can't determine the name. The downgrade's `op.drop_constraint(None, "table", type_="foreignkey")` will fail when Postgres actually runs it, because Postgres auto-names FK constraints and Alembic has no way to reverse-lookup the generated name at downgrade time.

**Fix**: **always give FK constraints explicit names** in the model:

```python
# backend/app/models/program_assignment.py
__table_args__ = (
    ForeignKey("program_teams.id", name="fk_program_assignments_program_team_id"),
)
```

Explicit names mean Alembic can cleanly drop them in the downgrade path. This is a one-time model hygiene cost that pays back every migration thereafter.

### Default FK `ON DELETE` is `NO ACTION` (effectively RESTRICT)

When a child FK has no `ON DELETE` clause, PostgreSQL defaults to `NO ACTION` — which behaves like `RESTRICT` for the common case. If you need nullable FK cleanup on parent delete (the "orphan child rows on parent delete" pattern), either:

- Add `ondelete="SET NULL"` on the FK column definition, OR
- Null out children in the service layer before calling `db.delete(parent)`.

Don't assume the migration will "just DTRT" — explicit is always better.

## Workarounds

**Problem**: Autogenerated migration contains drops/recreates of unrelated constraints.
**Fix**: Delete those ops from the generated migration file before committing. Keep only the ops that correspond to your feature's model changes.
**Why**: Alembic autogenerate is a diff tool, not a feature-aware tool. It surfaces all drift, including drift introduced by prior hand-edited migrations.

**Problem**: `alembic downgrade` fails with "no constraint named None".
**Fix**: Edit the downgrade function to use the actual constraint name from Postgres (`\d+ tablename` in psql to find it), OR name all constraints explicitly in models going forward.
**Why**: See "`drop_constraint(None, ...)` fails at runtime" above.

## Patterns We Use

- **Hand-review every autogenerated migration**. Never `git add` a migration without reading the full `upgrade()` and `downgrade()` functions.
- **Name every FK and unique constraint explicitly** in the model's `__table_args__` or as a `name=` kwarg on `ForeignKey(...)`/`UniqueConstraint(...)`.
- **Mirror existing migration style** when hand-editing: see `backend/alembic/versions/55b198d5e2a6_add_functional_manager_id.py` for the canonical pattern (pure `op.add_column` / `op.create_foreign_key` calls, with matching `downgrade()` in reverse order).
- **Run `make reset-db` after creating a new migration** to verify it applies cleanly on a fresh DB, not just as an upgrade from the previous revision.
